# OrchestrateMCP v0.1.0 — draft release notes

> **Draft:** do not publish or tag until every release gate below is checked
> against the exact release commit.

OrchestrateMCP is a read-only, stateless MCP server that turns an AI workflow
goal into an evidence-backed product plan. It is designed to say which parts of
a plan come from the registry, which parts are suggestions, where human
approval is required, and where coverage is incomplete.

## Included in the first release

- A deterministic registry of workflow components, relationships, routes,
  playbooks, stacks, and worker roles.
- Goal-to-product planning with connections, safety gates, build controls,
  hosting/monitoring choices, and implementation handoff artifacts.
- Coverage accounting, provenance tags, and release-trust floors that keep
  unsupported or untested parts visible.
- A reproducible seven-prompt registry conformance benchmark with stable input
  fingerprints and machine-readable required/forbidden-component results.
- Read-only MCP tools over local stdio and the hosted Streamable HTTP endpoint.
- Starter flows and setup guides for ChatGPT, Claude, Cursor, and Claude
  Desktop.
- Safe source-export tooling that excludes credentials, local settings,
  runtime state, and generated artifacts.

## Security and privacy boundary

OrchestrateMCP does not run the workflows it plans, store credentials, store
prompt or plan content, or make LLM calls inside its MCP tools. Credentials for
generated workflows remain the user's responsibility and belong in the target
platform's secret manager.

## Known limits

- Registry coverage is explicit but incomplete; a composed or suggested step
  is not equivalent to a tested integration.
- A generated plan or build brief is advisory and still requires review,
  implementation testing, and human approval for irreversible actions.
- The public demo evidence must label real, simulated, and stubbed downstream
  services; it must not imply production validation where none exists.
- DASH is a separate public companion and is not included as a finished
  monitoring application in v0.1.0.
- The automated benchmark does not measure LLM quality or vanilla-versus-MCP
  uplift; those comparisons require isolated client runs under the manual
  protocol.

## Release gate

- [ ] `pnpm install --frozen-lockfile` succeeds from a fresh clone.
- [ ] `pnpm build` succeeds before any other generation or verification command.
- [ ] `pnpm verify` passes on the exact release commit.
- [ ] `pnpm audit` reports no known dependency vulnerabilities.
- [ ] `health_check` counts and published/total terminology match README, SITE,
      and launch assets.
- [ ] ChatGPT, Claude, and Cursor first-plan checks pass.
- [ ] The public benchmark is reproducible and contains no private LAB data.
- [ ] The recorded demo labels real, simulated, and stubbed steps.
- [ ] Repository license, homepage, topics, security policy, and release notes
      are visible on GitHub.
- [ ] The Soft Launch Wave 0 review returns GO.
- [ ] Henrik approves the final release commit and copy.
